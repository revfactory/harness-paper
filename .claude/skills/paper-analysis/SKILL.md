# Paper Analysis — 논문 세부 분석 절차

분석의 가치는 **실제로 읽고, 하네스 렌즈로 구조화**하는 데 있다. 제목 추측 요약은 서베이를 오염시킨다.

## 왜 실제로 읽어야 하는가

대상 논문은 지식 컷오프 이후 발표되어 당신의 기억에 없다. 제목만으로 방법·결과를 추정하면 그럴듯하지만 틀린 요약이 나온다. 최소한 **초록 전문 + 도입/결론**을 WebFetch로 확인한 뒤 분석한다. 접근 불가 시 확인 가능한 범위만으로 제한 분석하고 그 사실을 명시한다.

## 하네스 렌즈

이 서베이의 관심은 "에이전트가 작업을 수행하도록 감싸는 스캐폴딩"이다. 각 논문에서 다음을 찾는다:
- 컨텍스트를 어떻게 구성·관리하는가 (context engineering)
- 툴 호출을 어떻게 조직하는가 (tool orchestration)
- 메모리·장기 과제를 어떻게 다루는가
- 에이전트 루프·제어 흐름 설계
- 평가·벤치마크 하네스 기여
- 멀티에이전트 조율 구조

## 분석 템플릿

논문 1건 → `_workspace/02_analysis/{paper-id}.md`:

```markdown
# {제목}

- 저자/기관: {...}
- 날짜: {YYYY-MM-DD}
- URL: {...}
- 확인수준: {전문 | 초록+도입/결론 | 초록만 | 접근불가}

## 한 문장 요약
## 문제 정의
## 핵심 방법
## 주요 기여
## 실험 결과 (정량)   ← 수치는 원문 그대로 + 출처(표/섹션)
## 한계               ← 저자 명시 + 분석자 판단 모두
## 하네스 엔지니어링 관련성
- 하위 주제 태그: {context-engineering | tool-orchestration | memory | agent-loop | eval-harness | multi-agent | 기타}
- 왜 중요한가:
## 인용용 핵심 사실 (검증 대상)   ← 리포트에 인용될 구체 주장. 검증자가 재확인 가능하게.
```

## 원칙

- **정량 결과는 원문 그대로.** 개선폭·점수를 과장하지 않고, 출처 위치를 남긴다.
- **한계를 반드시 찾는다.** 서베이의 "발전 방향" 절은 여기서 나온다. 저자가 밝힌 한계 + 방법론적 한계를 모두 기록.
- **택소노미 태그를 일관되게.** synthesizer가 분류를 귀납할 수 있도록 위 태그 어휘를 재사용한다.
- **인용용 사실은 검증 가능하게.** "SWE-bench에서 X% 달성" 처럼 검증자가 원문과 대조할 수 있는 구체 형태로 쓴다.

## 정직성

- 읽지 못한 부분을 추측으로 채우지 않는다. 확인수준을 정직하게 표기.
- 하네스와 무관하다고 판단되면 폐기하지 말고 "제외 권고 + 사유"를 남겨 리더가 판단하게 한다.
- 메타데이터 오류(날짜·저자) 발견 시 정정하고 검증자에게 알린다.
