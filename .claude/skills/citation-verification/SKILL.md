# Citation Verification — 적대적 인용 검증 절차

검증자의 임무는 통과시키는 것이 아니라 **가짜를 걸러내는 것**이다. 기본 태도는 "이 논문은 존재하지 않을 수도 있다"이다.

## 왜 적대적으로 검증하는가

대상 논문은 모두 지식 컷오프 이후 발표되었다. 정찰·분석 단계에서 그럴듯한 환각(존재하지 않는 논문, 실제와 다른 저자/날짜, 원문에 없는 수치)이 섞여 들어올 수 있다. 후보 기록을 신뢰하지 않고 **독립적인 검색**으로 처음부터 다시 확인한다.

## 검증 절차 (후보 1건당)

1. **실존 확인**: 제목으로 독립 검색 → 실제 arXiv/출판/블로그 페이지가 뜨는가? URL이 살아있는가(WebFetch 200)?
2. **식별자 대조**: arXiv ID 등이 실제로 그 논문을 가리키는가? (ID로 직접 접근해 제목 일치 확인)
3. **저자·소속 확인**: 후보에 적힌 저자/기관이 실제 페이지와 일치하는가?
4. **날짜 확인**: 실제 발표/submission 날짜 → 범위(2026-05~07) IN/OUT 판정
5. **주장 대조**: 분석의 "인용용 핵심 사실"의 수치·주장이 원문에 실제로 있는가?

## 환각 탐지 신호 (→ REJECTED)

- 제목으로 검색해도 어떤 학술 소스에도 안 나옴
- arXiv ID가 존재하지 않거나 전혀 다른 논문
- 저자·소속이 실존하지 않거나 논문과 무관
- URL이 404이거나 완전히 다른 내용
- "인용용 핵심 사실"의 수치가 원문에 없음 (날조/과장)

## 판정 어휘 — 구분이 중요하다

| 판정 | 의미 |
|------|------|
| **CONFIRMED** | 실존 확인 + 날짜 범위 내 + 인용 사실 일치 → 리포트 본문 진입 허용 |
| **OUT-OF-RANGE** | 실존하나 날짜가 범위 밖 → 본문 배제, 부록에서 맥락용 |
| **REJECTED** | 실존 부정 또는 날조 확인 → 완전 배제, 사유 기록 |
| **UNVERIFIED** | 확인 실패(접근 장애 등), 존재 부정과 다름 → 본문 배제, 사유 기록 |

**핵심 구분:** "존재하지 않음"(REJECTED)과 "확인하지 못함"(UNVERIFIED)은 다르다. 접근이 막혀 확인 못 한 것을 가짜로 단정하지 않는다. 반대로, 확인 못 한 것을 본문에 넣지도 않는다.

## 출력 스키마

`_workspace/03_verification/verdicts.md`:

```markdown
# 검증 결과

## 요약
- 총 후보: {n} | CONFIRMED: {n} | OUT-OF-RANGE: {n} | REJECTED: {n} | UNVERIFIED: {n}

## 판정 상세
### {paper-id} — {제목}
- 판정: {CONFIRMED | OUT-OF-RANGE | REJECTED | UNVERIFIED}
- 실존확인: {독립 검색으로 확인한 URL/소스}
- 날짜확인: {날짜} → 범위 {IN | OUT}
- 인용 사실 대조: {일치 | 불일치 항목}
- 근거: {판정 이유}
```

## 원칙

- **최소 2개 독립 소스로 교차.** 한 소스 접근 실패로 UNVERIFIED 단정하지 말고, arXiv·Semantic Scholar·Google Scholar 등을 교차한다.
- **부분 일치 주의.** 제목 유사·저자 다름은 별개 논문일 수 있다. 상세 대조 후 판정하고 근거를 남긴다.
- **의심스러우면 배제.** 독립 확인 못 한 항목은 본문에 넣지 않는다. 이 보수성이 서베이 신뢰성의 근간이다.
- **환각 발견 시 전파.** 가짜 후보는 다른 검색 축에도 있을 수 있으므로 scout에게 SendMessage로 알린다.
